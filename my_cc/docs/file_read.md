# Read 工具（`src/tools/file_read.py`）实现思路

> 对应 TS 源码：`claude-code-main/src/tools/FileReadTool/`（`FileReadTool.ts` / `prompt.ts` / `limits.ts`）

## 这个工具解决什么问题

模型要改代码、回答"这个文件里写了什么"，第一步永远是**读到磁盘上的真实文件内容**。在有这个工具之前，`QueryEngine` 里只有一个返回固定字符串的假工具（`FakeReadFileTool`），模型其实什么都读不到。`Read` 让模型能按绝对路径读取任意文本文件，并且为后续的 `Edit` 工具打好地基。

## 与 TS 源码的对应关系

TS 原版有 ~1184 行，绝大部分在处理多模态（图片 / PDF / Jupyter）、token 精确计数、去重缓存、技能发现、遥测等工业细节。我们只复刻了**文本读取主线**的核心决策，其余有意省略（见末节）。

## 关键设计决策

### 1. 参数与返回格式
- 工具名 `Read`，参数 `file_path`（必须是**绝对路径**）、可选 `offset` / `limit`。
- 返回 `cat -n` 风格：每行前面带右对齐行号 + 制表符。**这串行号不是给人看的好看，而是为 `Edit` 服务**——将来定位"改第几行"靠它。

### 2. 分页与两道大小防线
- 默认从第 1 行起最多读 `MAX_LINES_TO_READ = 2000` 行。
- 第一道防线：整文件读取时，先 `stat` 看大小，超过 `MAX_SIZE_BYTES = 256KB` **直接报错、根本不读**，提示改用 `offset/limit` 分段。这样不会因为一个巨型文件撑爆上下文。
- （TS 还有第二道防线：用 API 统计真实 token 数。需要联网，我们省略。）

### 3. `validate_input` 不碰磁盘
这是一条**安全原则**：参数校验阶段只做纯字符串检查（绝对路径校验、二进制后缀拒绝），把所有磁盘 I/O 推迟到**权限通过、真正 `call()`** 时再做。理由：在权限还没确认前就去摸文件系统是危险的。

### 4. 二进制文件直接拒绝
用一个后缀名集合（`.png/.pdf/.exe/.zip…`）粗判二进制，拒读。本工具只处理文本。

### 5. 友好的错误与边界处理
- 文件不存在 → 报错时顺带"是不是想找 xxx？"（`_find_similar`：同目录下找同名但扩展名/大小写不同的文件）。
- 空文件 / `offset` 越界 → 返回 `<system-reminder>` 警告，而不是返回空内容让模型困惑。

### 6. 写入 `read_file_state` 缓存（为 Edit 铺路）
每次读成功，把 `{内容, mtime, offset, limit}` 写进 `context.read_file_state[fp]`。
`Edit` 工具靠它判断"这个文件 AI 确实读过、且读完之后没被人改过"，才允许编辑。这是"读→改"闭环的关键一环。

### 7. 安全护栏
读完的内容后面附一段 `CYBER_RISK_REMINDER`：提醒模型可以分析恶意代码、但不得改进/增强它。

## 有意没做的部分

- 图片 / PDF / Jupyter notebook 的多模态读取（另外几百行）。
- 第二道 token 防线（需联网调 API 计数）。
- 去重 stub（同文件没变就返回占位以省 cache token）。
- 技能发现、遥测、macOS 截图替代路径等。
